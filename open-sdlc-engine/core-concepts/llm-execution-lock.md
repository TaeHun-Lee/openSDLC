# LLM Execution Lock

OpenSDLC를 도구 실행 환경을 가진 LLM에서 사용할 때는, 일반적인 생산성 최적화 습관과 OpenSDLC의 순차 실행 원칙이 충돌할 수 있습니다.

이 문서는 그 충돌을 방지하기 위한 모델 중립적 운영 메모입니다.

## 왜 필요한가

OpenSDLC는 문서 작성 규칙만 정의하는 시스템이 아닙니다.
에이전트의 **실행 순서**, **핸드오프 방식**, **검증 게이트**, **사용자 보고 방식**까지 포함하는 런타임 운영 모델입니다.

하지만 도구를 사용할 수 있는 LLM 실행 환경은 기본적으로 아래 성향을 가질 수 있습니다.

- 여러 파일을 병렬로 읽기
- 여러 단계를 한 번에 묶어 빠르게 처리하기
- 한 턴에서 끝까지 밀어붙이기
- 탐색과 실행을 동시에 압축하기

이런 습관은 일반적인 코딩 작업에는 유효할 수 있지만, OpenSDLC에서는 다음과 같은 비준수로 이어질 수 있습니다.

- 둘 이상의 에이전트가 동시에 활성화된 것처럼 보이는 현상
- Validator 게이트보다 구현 또는 테스트가 먼저 진행되는 현상
- 에이전트별 책임과 보고 주체가 흐려지는 현상
- 아티팩트 파이프라인은 있는 것처럼 보이지만 실제 실행은 병렬로 압축되는 현상

## 핵심 원칙

OpenSDLC가 활성화되면 다음 원칙을 강제합니다.

1. 특정 시점에는 오직 하나의 에이전트만 활성화된다.
2. 에이전트는 병렬로 실행되지 않는다.
3. 다음 단계는 이전 단계의 산출물과 필요한 검증 결과가 존재할 때만 시작된다.
4. 도구 효율보다 순차 실행 규칙이 우선한다.
5. 보고 형식뿐 아니라 실제 실행 전략도 순차적이어야 한다.

## LLM 환경에서의 해석 규칙

`Sequential Single-Agent Execution`은 단순한 서술 스타일 규칙이 아니다.
아래 모두에 적용되는 실행 제약이다.

- 에이전트 전환
- 파일 읽기 순서
- 툴 호출 방식
- 검증 게이트 통과 여부 확인
- 사용자에게 보이는 진행 보고

즉, OpenSDLC 실행 중에는 병렬 도구 사용이나 다중 단계 묶음 실행을 허용하지 않는다.

## 적용 위치

이 실행 잠금은 아래 3개 계층에 중복 반영되어 있다.

1. `open-sdlc-engine/prompts/system/initial-system-prompt-codex.md`
   - 부트스트랩 단계에서 `Codex execution guardrails`와 `Execution Lock`을 선언한다.
2. `open-sdlc-engine/prompts/agent/AgentCommon.txt`
   - 모든 에이전트가 공통으로 따라야 할 실행 잠금을 명시한다.
3. `open-sdlc-engine/prompts/agent/PMAgent.txt`
   - PMAgent가 실제 핸드오프와 사용자 상호작용을 통제할 때 순차 실행을 강제한다.

## 비준수 판정 기준

아래 중 하나라도 발생하면 OpenSDLC 프로세스 비준수로 본다.

- 둘 이상의 에이전트를 동시에 시작 또는 진행 중인 것처럼 보고함
- 병렬 도구 실행으로 여러 워크플로우 단계를 사실상 동시에 진행함
- Validator 결과 확인 전에 다음 에이전트로 넘어감
- 여러 에이전트 단계를 하나의 혼합 실행 블록으로 압축함
- 순차 실행 위반이 있었는데도 준수 완료로 보고함

## 실무 체크리스트

다음 에이전트로 넘기기 전에 항상 확인한다.

1. 현재 에이전트의 작업이 끝났는가
2. 필요한 아티팩트 파일이 실제로 생성되었는가
3. 필요한 Validator 결과가 존재하는가
4. 다음 단계로 넘어가는 것이 정책상 허용되는가
5. 이번 단계에서 병렬 실행 또는 혼합 보고가 없었는가

## 결론

OpenSDLC를 도구형 LLM 환경에서 안전하게 실행하려면, OpenSDLC를 단순한 프롬프트 템플릿이 아니라
**실행 스케줄링 규칙을 포함한 운영 모드**로 취급해야 한다.

순차 실행은 선택 사항이 아니라 준수 조건이다.
