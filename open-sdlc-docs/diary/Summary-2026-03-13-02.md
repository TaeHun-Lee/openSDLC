# OpenSDLC v1 재점검 요약

작성일: 2026-03-13  
차수: 02

## 1. 문서 목적

이 문서는 ValidatorAgent 도입과 관련 엔진 문서/프롬프트/템플릿 반영 이후, OpenSDLC v1 엔진을 기존 점검 기준으로 다시 검토한 결과를 정리한 재점검 기록이다.

이번 재점검의 초점은 다음 두 가지였다.

1. 전체 프로세스가 실제로 원활하게 이어질 수 있는가
2. 각 단계 산출물이 충분한 완전성과 무결성을 가지는가

## 2. 재점검 기준

이번 재점검은 아래 네 가지 기준으로 진행했다.

1. 프로세스 연결성
2. 산출물 완전성
3. 상태/판정/참조의 무결성
4. 종료 흐름의 자연스러움

## 3. 현재 엔진 상태 요약

현재 OpenSDLC v1 엔진은 초기 구조에 비해 한 단계 더 성숙한 상태로 평가된다.

- ValidatorAgent가 단계별 독립 검증 게이트로 추가되었다.
- 워크플로우가 `Req -> Val -> Code -> Val -> Test -> Val -> Coord -> Val -> PM` 구조로 재정렬되었다.
- ValidationReportArtifact가 추가되어 각 단계 검증 결과가 추적 가능한 산출물로 남게 되었다.
- PMAgent는 validator 결과를 무시하지 않는 상위 조정자 역할로 더 명확해졌다.
- FeedbackArtifact와 verification_report가 종료 우선순위와 validator 결과를 반영하도록 보강되었다.

## 4. 점검 결과

### 4.1 프로세스 연결성

프로세스 연결성은 전반적으로 양호하다.

- agent 정의, workflow, 공통 프롬프트, 개별 프롬프트, artifact 템플릿이 대부분 같은 흐름을 가리킨다.
- 전문 agent 산출물은 다음 전문 agent로 바로 전달되지 않고 ValidatorAgent를 거치도록 정리되었다.
- 재작업 경로도 `fail -> 원래 생성 agent 재작업` 방식으로 개념적으로는 잘 연결된다.

다만 요약 문서와 상세 문서 간 밀도 차이는 약간 남아 있다.

- 일부 상위 요약 문서는 VAL이 반복 등장하는 실제 상세 흐름을 축약해 표현하고 있다.
- 운영에는 큰 문제는 없지만, 개요 문서와 상세 workflow 문서의 표현 밀도를 장기적으로 더 맞추면 좋다.

### 4.2 산출물 완전성

산출물 완전성은 이전보다 확실히 좋아졌다.

- UC는 scope, constraints, acceptance criteria를 충분히 제공한다.
- IMPL은 traceability, files_changed, runtime_info를 제공한다.
- TEST는 defects, evidence, satisfaction_score를 제공한다.
- FB는 priority_tasks, done_criteria, completion_guidance를 포함한다.
- VAL은 validated_stage, blocking_reason, rework_target_artifact_id, action_rules까지 포함하게 되었다.

즉 각 artifact는 다음 단계 판단과 재작업 지시를 지원할 정도의 정보 밀도를 대체로 확보했다.

다만 ValidationReportArtifact의 `checks`는 단계별 필수 체크셋이 강제되지는 않는다.  
예를 들어 UC 검증과 TEST 검증이 동일 밀도로 작성되지 않을 가능성은 여전히 남아 있다.

### 4.3 무결성

무결성은 이전 점검 시점보다 가장 크게 개선된 영역이다.

- ValidationReportArtifact의 `status`를 `approved`에서 `actionable`로 조정해 `validation_result`와의 의미 충돌을 완화했다.
- `action_rules`를 추가하여 `pass/warning/fail`과 `proceed/rework/hold`의 조합을 명시했다.
- FeedbackArtifact에 `decision_rules`를 추가하여 점수, validator 결과, 상태값 사이의 관계를 문서화했다.
- 종료 우선순위를 workflow와 verification report에 명시해 암묵 규칙을 줄였다.

아직 남은 약점은 공통 상태값 체계가 여전히 기존 artifact 중심이라는 점이다.

- VAL에 대해 완전히 특화된 상태 체계를 새로 두지는 않았다.
- 그러나 현재 수준에서는 운영상 치명적 충돌보다는 표현상의 제약에 가깝다.

### 4.4 종료 흐름

종료 흐름도 실운영 가능한 수준으로 개선되었다.

- validator fail이 남아 있으면 종료할 수 없다는 우선순위가 명시되었다.
- completion recommendation은 validator 결과 이후에만 의미를 갖도록 정리되었다.
- 최종 종료는 반드시 사용자 승인 이후에만 가능하다는 원칙이 유지된다.
- verification_report 템플릿에 validator gate summary가 추가되어 PMAgent가 종료 권고의 근거를 더 명확히 제시할 수 있게 되었다.

여전히 남아 있는 경미한 보완 포인트는 완료 사례 표현의 풍부함이다.

- FeedbackArtifact는 이제 completion 경로를 표현할 수 있지만, 기본 예시 값은 여전히 반복 경로 중심이다.
- 장기적으로는 completion example을 별도로 하나 두면 종료 케이스 가독성이 더 좋아질 수 있다.

## 5. 종합 평가

현재 엔진은 다음과 같이 평가할 수 있다.

- 프로세스 연결성: 좋음
- 산출물 완전성: 좋음
- 무결성: 좋음에 가까운 보통 이상
- 종료 흐름: 실운영 가능한 수준

즉 현재 OpenSDLC v1 엔진은 "ValidatorAgent를 넣은 개념 제안" 단계를 넘어, 실제 운영 가능한 구조에 상당히 근접해 있다.

## 6. 남은 마무리 다듬기 포인트

현재 시점에서 남은 일은 구조적 재설계보다는 마무리 다듬기에 가깝다.

1. 요약 문서와 상세 문서 간 흐름 표현 밀도 맞추기
2. ValidationReportArtifact의 단계별 필수 체크셋 가이드 강화
3. FeedbackArtifact의 완료 사례 예시 보강

## 7. 결론

이번 재점검을 통해 확인된 핵심은 다음과 같다.

OpenSDLC v1 엔진은 이제 "LLM이 각 agent의 역할을 잘 지키길 기대하는 구조"에서 "역할 위반을 단계별 검증 게이트로 통제하는 구조"로 상당 부분 전환되었다.  
따라서 현재 남은 과제는 근본 구조 변경보다는 표현 밀도와 운영 편의성을 높이는 마무리 다듬기 단계로 보는 것이 적절하다.
