# OpenSDLC v1 분석 요약

작성일: 2026-03-13  
차수: 01

## 1. 문서 목적

이 문서는 사용자 설명과 `open-sdlc-engine`에 정의된 문서를 함께 분석하여, OpenSDLC v1의 구조, 운영 원칙, 에이전트 역할, 산출물 체계, 작업 규율을 누락 없이 정리한 요약본이다.

## 2. 프로젝트 정체성

OpenSDLC는 다중 AI Agents를 활용해 SDLC를 반복적으로 수행하는 Interactive AI SDLC 플랫폼의 시작 버전이다. 핵심 목표는 AI가 SDLC를 주도적으로 운영하되, 중요한 판단 지점에서는 사람과 상호작용하며 목표와 산출물을 계속 재정렬해 최종 결과물을 도출하는 것이다.

엔진 문서 기준으로 OpenSDLC는 SDLC 각 단계에서 생성되는 Artifact를 다음 에이전트의 실행 지침으로 사용하는 AI Software Factory 플랫폼이며, `Spiral Iteration` 구조를 통해 요구사항 정의, 구현, 검증, 피드백의 순환 고리를 반복한다.

## 3. 시스템 영역 구분

### 3.1 엔진(`open-sdlc-engine`)

- OpenSDLC의 개념, 설계 원칙, 워크플로우, 품질 게이트, 상태 값이 정의된다.
- 에이전트 종류와 각 역할, 책임, 입출력 산출물이 정의된다.
- 공통 프롬프트와 에이전트별 프롬프트가 정의된다.
- 아티팩트 템플릿과 검증 보고서 템플릿이 정의된다.
- 사용자 설명 기준으로 엔진 영역은 OpenSDLC의 단일 기준점이며, 일반 에이전트에게는 Read-Only 영역이다.
- 향후 특별한 엔진 관리 Agent만 엔진 수정 권한을 갖도록 설계되어 있다.

### 3.2 작업공간(`workspace`)

- 엔진을 사용해 실제 대상 프로젝트를 개발하는 공간이다.
- 각 프로젝트의 반복 개발 산출물과 실행 코드가 모두 여기에 저장된다.
- 엔진 정의상 기본 구조는 `workspace/{ProjectName}/iteration-{NN}/artifacts/`, `workspace/{ProjectName}/iteration-{NN}/verification_report.md`, `workspace/{ProjectName}/dev/` 이다.
- 모든 구현 결과는 작업공간에 남아야 하며, `dev/`는 항상 최신 실행 코드 상태를 유지해야 한다.

## 4. OpenSDLC의 핵심 설계 원칙

엔진 문서와 사용자 설명을 종합하면 다음 원칙이 시스템의 핵심이다.

1. `Artifact-Driven Execution`
   - 산출물은 단순 문서가 아니라 다음 에이전트의 실행 지시서다.
   - 모든 에이전트 출력은 구조화된 아티팩트여야 한다.

2. `Spiral Iteration`
   - 한 번에 완결하는 방식이 아니라 MVP부터 시작해 반복적으로 품질과 기능을 높인다.
   - 만족도 점수와 피드백을 기반으로 다음 이터레이션을 결정한다.

3. `Role-Based Orchestration`
   - PM, 요구사항, 구현, 테스트, 조정 역할이 명확히 분리된다.
   - 각 에이전트는 자신의 책임 범위 안에서만 동작한다.

4. `Human-Guided Development`
   - AI가 주도적으로 흐름을 운영하지만, 요구사항 확정과 프로젝트 종료 같은 중요 판단은 반드시 사용자 승인 하에 이뤄진다.
   - PMAgent는 사람과 에이전트 집단 사이의 유일한 인터페이스이자 통제탑이다.

5. `Strict Adherence`
   - 사용자가 지시하지 않은 임의 변경은 금지된다.
   - 디자인, 기능, 아이콘, 레이아웃, 기술 선택 모두 요청 범위 안에서만 수행해야 한다.

6. `Full Traceability`
   - 모든 구현과 테스트는 요구사항과 연결되어야 한다.
   - `source_artifact_ids`를 통해 산출물 간 연결 관계를 유지해야 한다.

7. `No-Shortcut Principle`
   - 여러 이터레이션을 자율 수행하더라도 `UC -> IMPL -> TEST -> FB -> verification_report` 흐름을 생략하거나 통합할 수 없다.
   - 각 단계는 독립된 물리적 파일로 남아야 한다.

8. `Detailed Reporting`
   - 에이전트 전환과 현재 작업 상황은 사용자가 추적 가능하도록 투명하게 보고되어야 한다.

## 5. 에이전트 구성과 책임

### 5.1 PMAgent

- 사용자 요청을 받아 전체 워크플로우를 시작한다.
- ReqAgent를 기동하고 이후 모든 에이전트 전환을 조율한다.
- 각 산출물이 템플릿과 규칙을 준수했는지 검증한다.
- 매 이터레이션마다 `verification_report.md`를 작성한다.
- 다음 반복이 필요한지 판단하고, 종료 전 반드시 사용자 승인 여부를 확인한다.

### 5.2 ReqAgent

- 사용자의 모호한 요청을 실행 가능한 `UseCaseModelArtifact`로 변환한다.
- 최소 실행 범위(MVP)를 기준으로 use case, acceptance criteria, scope, constraints, test focus를 정의한다.
- 다음 단계인 CodeAgent가 구현할 수 있도록 명확한 지시를 제공한다.

### 5.3 CodeAgent

- `UseCaseModelArtifact`를 기준으로 기능을 구현한다.
- 구현 내용, 변경 파일, 실행 정보, 한계 사항, 추적 관계를 `ImplementationArtifact`로 기록한다.
- 범위 밖 기능이나 임의 디자인 변경을 해서는 안 된다.

### 5.4 TestAgent

- `UseCaseModelArtifact`와 `ImplementationArtifact`를 함께 읽고 수용 기준 충족 여부를 검증한다.
- 테스트 결과, 결함, 개선 사항, 커버리지, 만족도 점수를 `TestReportArtifact`로 기록한다.
- 테스트는 반드시 acceptance criteria 기준으로 수행되어야 하며, 관찰 가능한 증거가 포함되어야 한다.

### 5.5 CoordAgent

- `TestReportArtifact`를 읽고 다음 반복 필요 여부를 결정한다.
- 만족도 90점 미만이면 다음 이터레이션을 위한 `FeedbackArtifact`를 생성한다.
- 만족도 90점 이상이면 완료 권고가 가능하지만, 실제 종료는 PMAgent를 통한 사용자 승인 이후에만 가능하다.

## 6. 산출물 체계와 의미

OpenSDLC의 표준 산출물은 다음 네 가지 YAML 아티팩트와 하나의 보고서로 구성된다.

1. `UseCaseModelArtifact (UC)`
   - 생성자: ReqAgent
   - 목적: 사용자 요구를 구현 가능하고 테스트 가능한 명세로 구조화
   - 핵심 항목: `user_story`, `use_cases`, `scope`, `constraints`, `test_focus`

2. `ImplementationArtifact (IMPL)`
   - 생성자: CodeAgent
   - 목적: 실제 구현 결과와 변경 사항을 기록
   - 핵심 항목: `implementation_target`, `implemented_features`, `files_changed`, `runtime_info`, `known_limitations`, `traceability`

3. `TestReportArtifact (TEST)`
   - 생성자: TestAgent
   - 목적: 수용 기준 기반 검증 결과와 결함, 만족도 기록
   - 핵심 항목: `test_results`, `defects`, `improvements`, `requirement_coverage`, `test_summary`

4. `FeedbackArtifact (FB)`
   - 생성자: CoordAgent
   - 목적: 테스트 결과를 다음 구현 지시로 변환
   - 핵심 항목: `feedback_reason`, `priority_tasks`, `prompt_for_code_agent`, `done_criteria`

5. `verification_report.md`
   - 생성자: PMAgent
   - 목적: 각 이터레이션의 규칙 준수 여부, 품질 상태, 후속 조치를 사용자에게 보고

## 7. 워크플로우와 종료 조건

표준 실행 흐름은 다음과 같다.

1. 사용자 입력
2. PMAgent가 워크플로우 시작
3. ReqAgent가 UC 생성
4. CodeAgent가 IMPL 생성 및 `dev/` 반영
5. TestAgent가 TEST 생성
6. CoordAgent가 FB 생성 또는 완료 권고
7. PMAgent가 검증 보고서 작성 및 사용자 확인 요청

종료 규칙은 다음과 같다.

- 만족도 점수 `< 90` 이면 자동으로 다음 이터레이션 대상이 된다.
- 만족도 점수 `>= 90` 이면 완료 가능한 수준으로 간주할 수 있다.
- 하지만 실제 프로젝트 종료는 반드시 사용자의 명시적 승인 이후에만 가능하다.

## 8. 에이전트가 반드시 지켜야 하는 운영 규율

사용자 설명과 엔진 프롬프트를 종합했을 때, 일반 에이전트는 아래 규율을 반드시 준수해야 한다.

1. 엔진 우선 원칙
   - 모든 작업은 엔진에 정의된 행동방식으로만 수행해야 한다.
   - 매 작업 시작 전 엔진에 정의된 개념, 규칙, 템플릿, 역할 지침을 확인해야 한다.
   - 특히 실제 산출물 생성 전에는 대응하는 물리적 템플릿 파일을 반드시 조회해야 한다.

2. 산출물 절대 준수 원칙
   - 산출물은 반드시 정의된 YAML 스키마 또는 보고서 템플릿을 그대로 따라야 한다.
   - 정의되지 않은 필드를 임의로 추가하면 안 된다.
   - 정보가 부족해도 필드를 삭제하지 않고 빈 값 또는 빈 리스트로 유지해야 한다.

3. 자체 검증 의무
   - 매 작업 완료 직후 결과물이 템플릿 필수 필드, 구조, 네이밍, 추적성 요구사항을 만족하는지 스스로 점검해야 한다.
   - PMAgent는 보고서 생성 전 선행 산출물 존재와 연결 관계를 검증해야 한다.

4. 증거주의와 데이터 무결성
   - 라인 번호, 파일 경로, 증거 항목은 추측으로 작성하면 안 된다.
   - 실제 파일을 다시 확인한 뒤 기록해야 한다.
   - PMAgent는 테스트/구현 증거 중 최소 두 건 이상을 교차 검증해야 한다.

5. 단계 생략 금지
   - 자율 반복 수행 상황에서도 에이전트 단계와 산출물 생성을 줄이거나 합치면 안 된다.

6. 투명한 보고
   - PMAgent는 각 단계 시작, 산출물 생성, 다음 단계 계획을 사용자에게 명확히 알려야 한다.

7. 범위 통제
   - 최소 구현을 우선하되 요청 범위를 벗어나지 않아야 한다.
   - 기존에 통과한 기능은 다음 반복에서 회귀되면 안 된다.

## 9. 품질 게이트와 상태 관리

엔진 문서에 정의된 주요 품질 게이트는 다음과 같다.

- 추적성 준수: 모든 결과물은 상위 아티팩트를 참조해야 한다.
- 회귀 금지: 이미 통과된 기능은 훼손되면 안 된다.
- 지시 준수: 요청하지 않은 변경은 결함으로 간주된다.

아티팩트 상태 값은 다음과 같이 정의되어 있다.

- `draft`
- `approved`
- `implemented`
- `feedback-required`
- `actionable`
- `completed`

## 10. 현재 분석에서 확인된 핵심 해석

1. OpenSDLC는 단순히 여러 에이전트가 협업하는 구조가 아니라, 아티팩트를 매개로 실행을 연결하는 규칙 기반 시스템이다.
2. 엔진은 단순 참고 문서가 아니라 모든 에이전트의 행동 상한선을 결정하는 운영 규범 저장소다.
3. 작업공간은 실제 개발 산출물이 축적되는 실행 영역이며, 엔진과 명확히 분리된다.
4. 사람은 단순 승인자에 그치지 않고, 중요한 의사결정 지점에서 방향을 재조정하는 상위 제어자 역할을 가진다.
5. OpenSDLC의 신뢰성은 자율성 자체보다도 템플릿 준수, 추적성, 증거 검증, 단계 생략 금지 같은 운영 규율에 의해 확보된다.

## 11. 결론

OpenSDLC v1의 본질은 "AI가 주도하되 사람의 승인과 재정렬을 내장한 반복형 SDLC 운영 체계"이다.  
이 체계에서 엔진은 불변에 가까운 기준과 규칙을 제공하고, 작업공간은 그 규칙을 적용해 실제 산출물을 누적하는 실행 영역이다.  
따라서 OpenSDLC를 구성하는 모든 Agent는 매 작업마다 엔진을 확인하고, 템플릿 기반 산출물을 생성하며, 작업 완료 후 스스로 검증하고, 필요한 지점에서는 반드시 사람과 상호작용해야 한다.

## 12. 회고: 역할 분리 신뢰성과 Validator 중심 운영 규칙

이번 검토에서 가장 크게 확인된 우려는 "LLM이 각 agent의 독립적 역할을 프롬프트만으로 안정적으로 지킬 수 있는가"였다.

검토 결과, 현재 OpenSDLC v1은 역할 정의와 아티팩트 파이프라인 자체는 비교적 명확하게 설계되어 있다. ReqAgent, CodeAgent, TestAgent, CoordAgent의 책임과 출력물은 잘 분리되어 있으며, 각 단계가 다음 단계의 입력이 되는 구조도 일관성이 있다.

하지만 실제 신뢰성은 프롬프트 강도보다 검증 장치의 강도에 더 크게 좌우된다는 점이 드러났다. 현재 엔진에는 self-verification, traceability, evidence cross-check 같은 좋은 규칙이 이미 들어 있으나, 상당 부분이 여전히 LLM의 자기 선언에 기대고 있다. 이 상태에서는 다음과 같은 리스크가 남는다.

1. CodeAgent가 구현 과정에서 요구사항 범위를 다시 해석하거나 확장할 수 있다.
2. TestAgent가 실제 검증보다 설명형 요약으로 흘러갈 수 있다.
3. CoordAgent가 defect를 실행 가능한 작업으로 충분히 변환하지 못할 수 있다.
4. PMAgent가 "검증 완료"를 선언하더라도 artifact 간 불일치가 남을 수 있다.

따라서 OpenSDLC의 다음 운영 기준은 "더 강한 프롬프트"보다 "더 강한 validator"가 되어야 한다는 결론에 도달했다. 이에 따라 엔진 문서에는 다음 원칙을 운영 규칙으로 승격했다.

- 스키마 검증 의무
- 추적성 검증 의무
- 증거 검증 의무
- 판정 일관성 검증
- validator 통과 결과를 요구하는 승인 게이트

이 판단은 OpenSDLC의 신뢰성을 "agent가 스스로 잘할 것이라는 기대"가 아니라 "역할 위반과 환각을 구조적으로 잡아내는 체계" 위에 올려야 한다는 방향 전환으로 이해할 수 있다.
