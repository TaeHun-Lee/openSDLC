# OpenSDLC v1 에이전트 운영 정비 및 PostItApp 진행 요약

작성일: 2026-03-13  
차수: 03

## 1. 문서 목적

이 문서는 오늘 세션에서 진행한 두 가지 축의 작업을 함께 정리하기 위한 기록이다.

1. OpenSDLC v1 엔진의 에이전트 운영 방식 정비
2. `PostItApp` 프로젝트의 실제 이터레이션 진행

이번 작업은 단순 코드 작성이 아니라, 엔진 문서와 실제 워크플로우가 함께 움직이는지 확인하면서 진행되었다는 점에서 의미가 있다.

## 2. 이번 세션에서 수행한 핵심 작업

### 2.1 에이전트 독립 구성 및 페르소나 정리

`open-sdlc-engine/agent-configs/` 폴더를 새로 만들고 아래 구성 파일들을 추가했다.

- `PMAgent.config.yaml`
- `ReqAgent.config.yaml`
- `ValidatorAgent.config.yaml`
- `CodeAgent.config.yaml`
- `TestAgent.config.yaml`
- `CoordAgent.config.yaml`
- `README.md`

이 작업을 통해 각 Agent는 다음 정보를 독립적으로 가지게 되었다.

- 역할 정의
- 입력/출력 범위
- handoff 규칙
- 페르소나
- 성공 조건

즉 기존의 프롬프트 파일 중심 구조에서 한 단계 더 나아가, Agent별 실행 설정 레이어가 추가되었다고 볼 수 있다.

### 2.2 사용자 인터랙션 경계 명확화

사용자와 직접 상호작용할 수 있는 Agent는 `PMAgent` 하나뿐이라는 원칙을 엔진 기준 문서에 명시적으로 반영했다.

반영한 주요 파일은 다음과 같다.

- `open-sdlc-engine/core-concepts/agent-definitions.md`
- `open-sdlc-engine/prompts/agent/AgentCommon.txt`
- `open-sdlc-engine/prompts/agent/PMAgent.txt`
- `open-sdlc-engine/agent-configs/PMAgent.config.yaml`
- `README.md`

이제 다른 Agent들은 다음만을 입력으로 사용해야 한다.

- PMAgent가 전달한 사용자 요청
- 상위 Artifact
- ValidationReportArtifact

이 정리는 실제 OpenSDLC 운영에서 역할 경계를 더 분명히 해 준다.

### 2.3 진행 보고 형식 실험과 운영 결정

세션 중간에 에이전트 보고를 1인칭 체크리스트형 형식으로 강제하는 방향을 실험적으로 문서에 반영했다.

다만 이후 사용자 피드백을 통해, 문서의 강제 형식보다 실제 대화는 더 자연스러운 이전 방식이 더 읽기 좋다는 점이 확인되었다.  
운영상으로는 다음과 같이 정리할 수 있다.

- 보고 형식 실험은 엔진 정비의 일부로 의미가 있었다.
- 실제 사용자 대화에서는 경직된 보고체보다 자연스러운 설명형 진행 보고가 더 적절했다.
- 따라서 향후에는 사용자 선호를 우선해 대화 톤을 조정하는 것이 바람직하다.

## 3. PostItApp 진행 내용

### 3.1 Iteration 01 완료

`PostItApp` 프로젝트에 대해 OpenSDLC 파이프라인 전체를 실제로 한 번 완주했다.

생성된 산출물은 다음과 같다.

- `UC-01.yaml`
- `UC-01-VAL-01.yaml`
- `IMPL-01.yaml`
- `IMPL-01-VAL-01.yaml`
- `TEST-01.yaml`
- `TEST-01-VAL-01.yaml`
- `FB-01.yaml`
- `FB-01-VAL-01.yaml`
- `verification_report.md`

그리고 실제 `workspace/PostItApp/dev/` 아래에 브라우저 기반 포스트잇 앱 초기 버전을 구현했다.

Iteration 01에서 구현한 핵심 기능:

- 포스트잇 추가
- 포스트잇 즉시 표시
- 포스트잇 삭제
- 빈 입력 예외 처리

테스트 결과 만족도는 `100`점으로 기록되었고, `CoordAgent`는 완료 권고 상태를 생성했다.  
다만 OpenSDLC 규칙상 프로젝트 종료는 자동이 아니라 사용자 승인 전제라는 점도 워크플로우상 확인되었다.

### 3.2 Iteration 02 착수

이후 사용자가 추가 개선으로 `포스트잇 수정 기능`을 요청했다.

이에 따라 `iteration-02`를 시작했고, 다음 단계까지 진행했다.

- `UC-02.yaml` 작성
- `UC-02-VAL-01.yaml` 검증 통과
- `IMPL-02.yaml` 작성
- `IMPL-02-VAL-01.yaml` 검증 통과

현재 구현 상태에서는 다음 기능이 추가되었다.

- 각 카드에서 `수정` 버튼으로 수정 모드 진입
- 수정 중 `저장`과 `취소`
- 빈 수정 입력 차단
- 기존 추가/삭제 흐름 유지

즉 Iteration 02는 구현 검증 단계까지 완료되었고, 다음은 `TestAgent` 단계부터 이어가면 된다.

## 4. 이번 세션의 의미

이번 세션은 단순히 앱 기능 하나를 더 만든 작업이 아니었다.

실제로는 다음 두 가지를 동시에 검증한 세션이었다.

1. OpenSDLC v1 엔진이 문서 수준을 넘어 실제 프로젝트 진행에 적용 가능한가
2. 엔진 운영 규칙과 사용자 상호작용 방식이 현실적인 사용 흐름과 잘 맞는가

결과적으로 다음 사실을 확인했다.

- OpenSDLC 파이프라인은 실제 프로젝트에 적용 가능하다.
- Validator 기반 단계 게이트는 충분히 유효하다.
- PMAgent 단일 사용자 인터페이스 원칙은 매우 중요하다.
- 대화 방식은 엔진 규칙 못지않게 사용자 체감 품질에 큰 영향을 준다.

## 5. 현재 상태 요약

현재 기준 상태는 다음과 같다.

- Agent 설정: 독립 구성 및 페르소나 정의 완료
- PostItApp Iteration 01: 완료 권고 상태
- PostItApp Iteration 02: 구현 및 구현 검증 완료, 테스트 단계 대기

## 7. 결론

오늘 세션을 통해 OpenSDLC v1은 문서 중심 개념을 넘어, 실제 에이전트 역할 분리와 단계 검증을 수행하는 운영 구조로 한층 구체화되었다.  
동시에 `PostItApp`은 단순 CRUD 수준에서 수정 기능까지 포함하는 방향으로 확장되었고, 실제 이터레이션 산출물이 누적되는 형태를 확인할 수 있었다.

즉 이번 작업은 엔진과 프로젝트 양쪽 모두에서 "구조가 실제로 굴러가기 시작했다"는 점을 보여주는 진전이라고 정리할 수 있다.
