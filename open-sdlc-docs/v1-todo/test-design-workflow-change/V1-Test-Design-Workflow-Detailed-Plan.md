# OpenSDLC v1 Test Design Workflow Change Detailed Plan

## 1. 문서 목적

이 문서는 OpenSDLC v1의 테스트 워크플로우를

```text
UC -> IMPL -> TEST
```

에서

```text
UC -> TEST 시나리오 생성 -> IMPL -> TEST 시나리오 실행 및 검증
```

구조로 재편하는 실험을 실제로 어떻게 진행할지 상세하게 정리한 작업 계획 문서다.

이 문서는 다음 대화를 직접적인 출발점으로 삼는다.

- 사용자 질문:
  - "테스트 디자인 워크플로우 변경을 진행하기 전에 확인할 사항: 너(Codex)의 도움을 받아 이 작업을 진행한다고 했을때, 최대한 세분화 하여 단계적으로 진행하는것과 전체 작업을 일괄로 진행하는 것의 장단점"
- Codex 답변:
  - 이번 작업은 구조 개편 성격이 강하므로, 최대한 세분화해서 단계적으로 진행하는 것이 더 안전하며, 의사결정 단위는 세분화하고 구현 단위는 묶어서 진행하는 방식이 가장 적절하다고 제안함

이 문서는 [README.md](../../v2-proposal/README.md)의 하위 실행 계획 문서 역할을 한다.

## 2. 왜 단계적 진행이 필요한가

이번 변경은 단순 문장 수정이나 단일 프롬프트 조정이 아니다.

영향 범위는 최소한 아래를 포함한다.

- 워크플로우 정의 문서
- 에이전트 책임 정의
- 프롬프트 구조
- 템플릿 스키마
- Validator 판정 기준
- 실제 예제 검증 방식

따라서 전체를 한 번에 변경하면 다음 문제가 생길 수 있다.

- 어떤 판단이 잘못됐는지 분리하기 어렵다
- 어떤 문서/프롬프트/스키마가 어긋났는지 추적하기 어렵다
- Codex가 연결성을 높이려다 자의적 해석을 섞을 위험이 커진다
- 실험 결과가 "무엇 때문에 좋아졌는지" 평가하기 어려워진다

반대로 단계적으로 진행하면 다음이 가능하다.

- 각 단계의 의사결정 근거를 명확히 기록할 수 있다
- 중간 검토와 방향 수정이 가능하다
- 실험성과를 단계별로 검증할 수 있다
- 영향 범위를 통제한 상태로 점진 적용할 수 있다

## 3. 진행 원칙

이번 작업은 아래 원칙으로 진행하는 것이 좋다.

### 3.1 의사결정 단위는 세분화한다

한 번에 "전부 바꾸는 것"보다 아래 질문을 분리해서 합의한다.

- 워크플로우를 정말 나눌 것인가
- 신규 아티팩트가 필요한가
- TestAgent를 분리할 것인가, 모드 분리만 할 것인가
- Validator는 무엇을 새로 검사해야 하는가
- v1 실험 범위를 어디까지로 제한할 것인가

### 3.2 구현 단위는 적절히 묶는다

문서 하나 수정하고 멈추는 식으로 너무 잘게 쪼개면 피로도가 커진다.

따라서 아래처럼 한 덩어리씩 진행하는 것이 적절하다.

- 워크플로우 문서 묶음
- 아티팩트 스키마 묶음
- 프롬프트/에이전트 정의 묶음
- Validator 규칙 묶음
- 검증 시뮬레이션 묶음

### 3.3 먼저 "실험 가능한 최소 변경"을 정의한다

v1 전체를 재설계하는 것이 아니라, 먼저 다음 질문에 답할 수 있을 만큼만 바꾼다.

- TEST-DESIGN 단계를 넣으면 실제로 테스트 누락이 줄어드는가
- `main_flow`, `preconditions`, `trigger` 누락을 더 잘 잡을 수 있는가
- 기존 사례에서 개선 효과가 명확히 보이는가

## 4. 권장 작업 순서 개요

전체 작업은 아래 7단계로 진행하는 것을 권장한다.

1. 워크플로우 개정안 확정
2. 신규/변경 아티팩트 구조 확정
3. 에이전트 책임과 프롬프트 개정
4. Validator 판정 기준 개정
5. 예시 케이스 기반 산출물 시뮬레이션
6. 실제 v1 문서 반영
7. 소규모 검증 실행 및 평가

## 5. 단계별 상세 계획

## 5.1 1단계: 워크플로우 개정안 확정

### 작업 목표

v1에서 `TEST`를 1단계로 유지할지, `TEST-DESIGN`과 `TEST-EXECUTION`으로 나눌지 구조를 명확히 확정한다.

### 세부 작업

- 기존 워크플로우 문서의 현재 파이프라인 정의 정리
- 신규 파이프라인 초안 작성
- 각 단계의 입출력 연결 관계 정의
- Validation 게이트 위치 재배치
- PMAgent handoff 규칙 영향 검토

### 산출물

- 개정된 파이프라인 초안
- 단계별 handoff 표
- 신규 단계 추가에 따른 영향 요약

### 완료 효과

- 전체 변경의 골격이 확정된다
- 이후 아티팩트와 프롬프트 개정이 일관된 기준 위에서 진행된다

### 주의점

- 여기서 아직 프롬프트와 템플릿까지 동시에 수정하지 않는다
- 먼저 "어떤 구조로 갈지"만 합의한다

### 다음 단계

- 2단계: 신규/변경 아티팩트 구조 확정

## 5.2 2단계: 신규/변경 아티팩트 구조 확정

### 작업 목표

새 워크플로우를 지탱할 아티팩트 구조를 정한다.

### 세부 작업

- `TestDesignArtifact` 신규 도입 여부 확정
- `TestReportArtifact`를 실행/검증 중심으로 재정의
- `ValidationReportArtifact`에 TEST-DESIGN 전용 체크 항목 추가 방향 정리
- 신규 필드 정의
  - `main_flow_coverage`
  - `precondition_coverage`
  - `trigger_coverage`
  - `observable_outcome_coverage`
  - `runtime_evidence_required`

### 산출물

- 신규 템플릿 초안
- 기존 템플릿 변경안
- 필드별 목적 설명

### 완료 효과

- 이후 프롬프트가 어떤 형식으로 산출물을 작성해야 하는지 기준이 생긴다
- Validator가 무엇을 검사해야 하는지 구조가 보인다

### 주의점

- v2 수준의 과도한 확장보다, v1 실험에 필요한 최소 필드 위주로 시작한다

### 다음 단계

- 3단계: 에이전트 책임과 프롬프트 개정

## 5.3 3단계: 에이전트 책임과 프롬프트 개정

### 작업 목표

새 워크플로우에 맞게 Test 관련 책임을 재정의한다.

### 세부 작업

- TestAgent 분리 여부 결정
  - `TestDesignAgent` 신규 도입
  - 또는 기존 `TestAgent`의 Design/Execution 모드 분리
- CodeAgent가 TEST-DESIGN을 입력으로 받도록 조정
- PMAgent handoff 문구 및 단계 설명 수정
- ValidatorAgent의 TEST-DESIGN 검증 역할 추가

### 산출물

- 에이전트 책임 매트릭스
- 프롬프트 변경안
- 입력/출력 관계 표

### 완료 효과

- 실제 운영 시 누가 무엇을 해야 하는지가 명확해진다
- "설계"와 "실행"이 문서뿐 아니라 역할 수준에서도 분리된다

### 주의점

- TestAgent를 아예 둘로 나누면 영향 범위가 커진다
- v1 실험 단계에서는 "하나의 에이전트, 두 모드" 방식이 더 현실적일 수 있다

### 다음 단계

- 4단계: Validator 판정 기준 개정

## 5.4 4단계: Validator 판정 기준 개정

### 작업 목표

새 구조에서 무엇을 통과시키고 무엇을 막을지 더 구체적으로 정의한다.

### 세부 작업

- TEST-DESIGN 단계 필수 체크 정의
  - AC coverage
  - main flow coverage
  - precondition coverage
  - trigger coverage
  - observable outcome coverage
- TEST-EXECUTION 단계 필수 체크 정의
  - 설계된 시나리오 실행 여부
  - evidence completeness
  - runtime/code evidence 구분
- 누락 시 `warning`/`fail` 판단 규칙 정리

### 산출물

- Validator 체크리스트 초안
- ValidationReportArtifact 확장안
- 단계별 `pass/warning/fail` 기준표

### 완료 효과

- 테스트 설계가 부실해도 그냥 넘어가는 일이 줄어든다
- "테스트가 있었다"가 아니라 "충분한 테스트 설계가 있었다"를 검증할 수 있다

### 주의점

- 기준을 너무 세게 두면 초기 실험에서 전부 fail이 날 수 있다
- v1 실험은 `warning` 중심으로 시작하고, 효과가 보이면 강화하는 방식도 가능하다

### 다음 단계

- 5단계: 예시 케이스 기반 산출물 시뮬레이션

## 5.5 5단계: 예시 케이스 기반 산출물 시뮬레이션

### 작업 목표

개정 구조가 실제로 어떻게 동작하는지 사례를 통해 확인한다.

### 세부 작업

- `TetrisLikeGame` 사례를 기준으로 신규 아티팩트 시뮬레이션
- `UC-01` 또는 유사한 요구사항에서 `TEST-DESIGN` 산출물 초안 생성
- 해당 설계 기준으로 `IMPL`과 `TEST-EXECUTION`이 어떻게 달라지는지 비교
- 기존 `TEST-01` 누락과 새 설계 결과 비교

### 산출물

- 샘플 `TestDesignArtifact`
- 샘플 `TestExecution/TestReportArtifact`
- 기존 방식 vs 개정 방식 비교표

### 완료 효과

- 이론이 아니라 실제 개선 효과를 눈으로 확인할 수 있다
- 문서 개정 전에 문제점과 장점을 검증할 수 있다

### 주의점

- 이 단계는 실험용 문서로 남기고, 아직 운영 기준 문서에 즉시 반영하지 않아도 된다

### 다음 단계

- 6단계: 실제 v1 문서 반영

## 5.6 6단계: 실제 v1 문서 반영

### 작업 목표

검증된 구조를 실제 v1 문서와 템플릿에 반영한다.

### 세부 작업

- `workflow.md` 수정
- `core-concept.md` 수정
- `agent-definitions.md` 수정
- `TestAgent.txt` 수정 또는 신규 프롬프트 추가
- 템플릿 추가/수정

### 산출물

- 반영된 문서 세트
- 신규/변경 템플릿 세트
- 변경 요약 문서

### 완료 효과

- 실험이 더 이상 메모 수준이 아니라 실제 v1 운영 규칙이 된다

### 주의점

- 이 단계에서는 변경 범위가 커지므로 커밋을 논리 단위로 나누는 것이 좋다

### 다음 단계

- 7단계: 소규모 검증 실행 및 평가

## 5.7 7단계: 소규모 검증 실행 및 평가

### 작업 목표

문서 반영 후 실제 운영 효과를 확인한다.

### 세부 작업

- 기존 문제 사례 재실행
- 신규 요구사항 1건으로 파일 생성 시뮬레이션
- TEST-DESIGN 단계가 실제 누락을 줄였는지 평가
- validator 경고/실패 패턴 분석

### 산출물

- 검증 리포트
- 개선 효과 요약
- 추가 수정 필요 항목 목록

### 완료 효과

- 구조 개편이 실제로 유효한지 확인할 수 있다
- 이후 v2 proposal과 연결할 실증 자료가 생긴다

### 주의점

- 문서만 바뀌고 실제 품질이 안 좋아질 수도 있으므로, 반드시 사례 기반 평가가 필요하다

## 6. 단계적 진행과 일괄 진행 비교 정리

## 6.1 최대한 세분화하여 단계적으로 진행할 때

### 장점

- 의사결정이 명확하다
- 중간 수정이 쉽다
- 영향 범위를 통제하기 좋다
- Codex가 자의적 연결을 덜 하게 된다
- 검증 결과를 단계별로 비교하기 쉽다

### 단점

- 시간이 더 걸린다
- 문서와 커밋 수가 늘어난다
- 중간 산출물이 많아져 관리 부담이 생긴다

## 6.2 전체 작업을 일괄로 진행할 때

### 장점

- 빠르게 전체 구조를 볼 수 있다
- 초기 완성형 초안을 한 번에 만들기 쉽다

### 단점

- 설계 오류의 원인 추적이 어렵다
- 문서/프롬프트/템플릿/validator가 동시에 바뀌며 정합성 관리가 어려워진다
- Codex가 비어 있는 연결을 자의적으로 보간할 위험이 커진다

## 6.3 이번 작업에 대한 권장 결론

이번 변경은 "완전 세분화"나 "완전 일괄"보다 아래 방식이 가장 적절하다.

- 의사결정은 세분화한다
- 구현은 논리적 묶음 단위로 진행한다
- 실제 v1 반영 전에는 반드시 샘플 시뮬레이션을 거친다

즉, 아래 전략을 권장한다.

```text
워크플로우 확정
-> 아티팩트 구조 확정
-> 프롬프트/에이전트 개정
-> validator 규칙 개정
-> 샘플 검증
-> 실제 반영
-> 운영 평가
```

## 7. 권장 커밋 전략

실제 작업 시에는 아래처럼 커밋을 나누는 것이 좋다.

1. 워크플로우/개념 문서 변경
2. 테스트 설계 아티팩트 및 템플릿 추가
3. 에이전트 프롬프트 개정
4. validator 규칙 개정
5. 샘플/검증 문서 추가

이렇게 하면 나중에 어떤 변경이 어떤 효과를 냈는지 추적하기 쉽다.

## 8. Codex 제안 결론

Codex가 권장하는 방식은 다음과 같다.

- 이번 변경은 단계적으로 진행한다
- 다만 너무 잘게 쪼개지 않고, "의사결정 묶음" 단위로 진행한다
- 실제 v1 반영 전에 `TetrisLikeGame` 같은 기존 실패 사례로 샘플 검증을 수행한다
- 검증에서 효과가 확인되면 그 다음에야 v1 운영 문서에 공식 반영한다

즉, 이번 작업은 "바로 전체 반영"보다

```text
설계 -> 샘플 검증 -> 공식 반영
```

순서로 가는 것이 가장 안전하고 효과적이다.
