# Step 07 - Sample Simulation

## 단계 대상 작업

- 작은 대표 요구사항을 기준으로 기존 v1 방식과 개정 방식 비교 시뮬레이션
- 테스트 설계 누락이 실제로 어떻게 드러나는지 문서화
- 이후 운영 문서 반영 전에 실효성 판단 근거 확보

## 작업 진행 계획

- 입력 상태, 시간 경과, 관찰 가능한 결과가 모두 포함된 작은 요구사항을 하나 선정한다.
- 동일 요구사항에 대해 아래 두 흐름을 비교한다.
  - 기존 v1: `UC -> IMPL -> TEST`
  - 개정안: `UC -> TEST-DESIGN -> IMPL -> TEST-EXECUTION`
- 각 흐름에서 어떤 항목이 구조적으로 강조되거나 누락되는지 정리한다.
- 마지막에 실험 효과와 다음 반영 판단 근거를 요약한다.

## 기대 효과

- 문서 개정 전에 구조 변경의 실효성을 구체적으로 확인할 수 있다.
- 어떤 항목이 기존 방식에서 놓치기 쉬운지 명확해진다.
- 실제 운영 문서 반영 시 우선순위를 더 정확히 잡을 수 있다.

## 예상 변경 범위

- 이번 단계는 시뮬레이션 문서와 이력 정리에 집중한다.
- 실제 운영 문서, 프롬프트, 템플릿 본문은 아직 수정하지 않는다.

## 시뮬레이션 대상 요구사항

### 샘플 User Story

> 사용자는 웹 페이지에서 시작 버튼을 누르면 카운터가 1초마다 1씩 증가하는 타이머를 볼 수 있어야 한다. 일시정지 버튼을 누르면 증가가 멈추고, 다시 시작하면 이어서 증가해야 한다.

### 이 요구사항을 선택한 이유

- `trigger`: 시작 버튼 클릭
- `precondition`: 초기 로드 후 버튼 조작 가능 상태
- `main_flow`: 시작 -> 시간 경과 -> 숫자 증가
- `alternate_flow`: 일시정지 -> 재시작
- `observable_outcome`: 숫자가 1초마다 증가, 일시정지 시 멈춤

즉, 이번 실험에서 확인하려는 핵심 요소를 모두 포함한다.

## 1. 기존 v1 방식 시뮬레이션

흐름:

```text
UC -> IMPL -> TEST
```

### 1.1 예상 UC 특징

- acceptance criteria 중심으로 다음 정도는 정리될 가능성이 높다.
  - 시작 버튼이 보인다
  - 카운터 숫자가 표시된다
  - 일시정지 버튼이 보인다

### 1.2 예상 IMPL 특징

- 버튼 2개와 카운터 UI는 구현될 가능성이 높다
- 타이머 증가 로직도 구현될 수 있다
- 하지만 다음 항목은 구현 상세에 따라 어긋날 위험이 있다.
  - 첫 시작 전 초기 상태
  - 일시정지 후 실제 증가 정지 여부
  - 재시작 시 카운터가 리셋되지 않고 이어지는지 여부

### 1.3 예상 TEST 특징

기존 방식에서는 테스트가 다음처럼 AC 중심으로 수렴하기 쉽다.

- 시작 버튼 표시 확인
- 숫자 영역 표시 확인
- 일시정지 버튼 표시 확인
- 카운터 값이 증가하는 코드 존재 확인

### 1.4 기존 방식에서 놓치기 쉬운 항목

- 페이지 초기 로드 직후 버튼이 실제로 동작 가능한 상태인지
- 시간 경과 기반 증가가 실제로 1초 간격으로 일어나는지
- 일시정지 후 증가가 정말 멈추는지
- 재시작 시 이전 값에서 이어지는지
- 타이머 중복 실행으로 숫자가 2배 속도로 증가하지 않는지

### 1.5 예상 Validator 한계

- `acceptance_criteria_mapping`은 통과할 수 있다
- evidence가 코드 기반으로만 제출돼도 형식상 통과 가능성이 있다
- 하지만 실제 시간 기반 동작 누락은 구조적으로 약하게 드러난다

## 2. 개정 방식 시뮬레이션

흐름:

```text
UC -> TEST-DESIGN -> IMPL -> TEST-EXECUTION
```

### 2.1 예상 TEST-DESIGN 특징

여기서는 최소 아래 시나리오가 분리되어야 한다.

#### 시나리오 A - 초기 상태 확인

- setup: 페이지 최초 로드
- action: 아무 버튼도 누르지 않음
- expected_result: 카운터는 초기값 유지, 버튼이 조작 가능 상태
- evidence_type: `runtime` 또는 `mixed`

#### 시나리오 B - 시작 후 1초 경과 증가 확인

- setup: 페이지 최초 로드
- action: 시작 버튼 클릭 후 1초 이상 대기
- expected_result: 카운터가 1 증가
- evidence_type: `runtime`

#### 시나리오 C - 일시정지 동작 확인

- setup: 카운터가 증가 중인 상태
- action: 일시정지 버튼 클릭 후 1초 이상 대기
- expected_result: 카운터 값이 더 이상 증가하지 않음
- evidence_type: `runtime`

#### 시나리오 D - 재시작 이어서 증가 확인

- setup: 일시정지 직후 상태
- action: 다시 시작 버튼 클릭 후 1초 이상 대기
- expected_result: 이전 값 다음 숫자부터 다시 증가
- evidence_type: `runtime`

### 2.2 CodeAgent에 생기는 차이

CodeAgent는 단순히 버튼과 증가 함수만 만드는 것이 아니라, 아래를 구현해야 한다는 압박을 받는다.

- 시작 전/후 상태 전이
- 중복 interval 생성 방지
- 일시정지 시 interval 정지
- 재시작 시 이어서 증가

즉, 테스트 기준이 먼저 고정되므로 구현이 더 구체적인 동작에 묶인다.

### 2.3 TEST-EXECUTION에서 생기는 차이

실행 단계에서는 이미 설계된 시나리오를 따라야 하므로 다음을 명확히 기록할 수 있다.

- 어떤 시나리오가 실행되었는지
- runtime evidence가 실제 제출됐는지
- 어떤 동작이 실패했는지
- 설계된 시나리오 중 미실행 항목이 있는지

### 2.4 Validator에서 생기는 차이

`TEST-DESIGN` 단계:
- 시간 경과 기반 증가 시나리오가 빠졌으면 `fail`
- 일시정지/재시작 흐름이 빠졌으면 `warning` 또는 `fail`
- runtime evidence required를 code-only로 설계했으면 `fail`

`TEST-EXECUTION` 단계:
- 시나리오 D를 실행하지 않았다면 `scenario_execution_completeness` 실패 가능
- runtime evidence 없이 코드 설명만 제출하면 `evidence_type_consistency` 실패 가능

## 3. 기존 방식 vs 개정 방식 비교 요약

| 비교 항목 | 기존 v1 | 개정안 |
| :--- | :--- | :--- |
| 테스트 설계 시점 | 구현 후 | 구현 전 |
| AC 커버리지 | 상대적으로 보장됨 | 보장됨 |
| `main_flow` 커버리지 | 누락 가능 | 구조적으로 요구됨 |
| `preconditions` 커버리지 | 누락 가능 | 구조적으로 요구됨 |
| `trigger` 커버리지 | 누락 가능 | 구조적으로 요구됨 |
| `observable_outcome` 검증 | 약할 수 있음 | 설계 단계에서 강제 가능 |
| runtime evidence 요구 | 약함 | 시나리오별 명시 가능 |
| validator 누락 탐지력 | 제한적 | 향상 가능 |

## 4. 시뮬레이션 결론

- 기존 방식은 버튼 존재, 코드 존재, 표면적 AC 충족에 머무를 가능성이 높다.
- 개정 방식은 시간 경과, 상태 전이, 재시작 동작 같은 실제 사용자 체감 시나리오를 더 강하게 드러낸다.
- 특히 `runtime evidence required`를 설계 단계에서 명시하는 점이 큰 차이를 만든다.

## 5. 현 단계 판단

- 샘플 시뮬레이션 기준으로 개정안은 기존 방식보다 테스트 누락을 더 구조적으로 드러낼 가능성이 높다.
- 따라서 다음 단계는 실제 운영 문서 반영 준비로 넘어가도 무리가 없다고 판단한다.
