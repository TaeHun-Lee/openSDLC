# Step 06 - Simulation Or Reflection Planning

## 단계 대상 작업

- 개정 워크플로우를 검증할 샘플 시뮬레이션 범위 정리
- 우선 적용 대상 사례 선정 기준 정리
- 사례 부재 시 진행 가능한 대체 검증 방식 정리

## 작업 진행 계획

- 제안 문서에서 추천한 `TetrisLikeGame` 사례를 우선 대상으로 검토한다.
- 현재 작업공간에서 실제 비교 가능한 사례가 존재하는지 확인한다.
- 사례가 없으면, 샘플 산출물 기반 시뮬레이션으로 전환하는 최소 검증 계획을 정리한다.
- 비교 실험 시 필요한 산출물 세트와 성공 기준을 정리한다.

## 기대 효과

- 실제 문서 반영 전에 어떤 방식으로 실효성을 검증할지 명확해진다.
- 사례가 없어도 작업이 멈추지 않도록 대체 경로가 생긴다.
- 다음 단계에서 바로 문서 반영으로 갈지, 샘플 시뮬레이션을 먼저 할지 결정할 수 있다.

## 예상 변경 범위

- 이번 단계는 시뮬레이션/검증 계획 수립과 이력 기록에 집중한다.
- 실제 운영 문서 수정은 아직 하지 않는다.
- 작업 이력 문서만 신규 생성한다.

## 작업 결과

### 1. 우선 적용 대상 기준

가장 좋은 비교 대상은 아래 특성을 가진 사례다.

- 기존 v1 방식으로 이미 `UC`, `IMPL`, `TEST`가 존재했던 사례
- 테스트 누락이 실제로 드러난 사례
- `main_flow`, `preconditions`, `trigger`, `observable_outcome` 누락 여부를 비교할 수 있는 사례
- runtime evidence가 필요한 동작을 포함한 사례

문서 기준 우선 추천 대상:

- `workspace/TetrisLikeGame`

이유:
- 입력 가능 전제조건
- 시간 흐름 기반 동작
- 구현은 존재하지만 사용자 체감상 실패 가능한 상호작용
- AC 중심 테스트의 한계가 잘 드러나는 사례

### 2. 현재 작업공간 상태 확인 결과

- 현재 `workspace`는 비어 있다.
- 따라서 문서에서 언급된 `workspace/TetrisLikeGame`를 즉시 읽어 비교 실험을 수행할 수는 없다.

이 상태에서 가능한 검증 경로는 두 가지다.

### 3. 권장 검증 경로 A - 기존 사례 확보 후 비교 실험

조건:
- `TetrisLikeGame` 또는 동급의 기존 v1 산출물 사례를 확보할 수 있을 때

진행 방식:

1. 기존 v1 산출물 세트 확보
   - `UC`
   - `IMPL`
   - `TEST`
   - 관련 validator 결과

2. 동일 요구사항 기준으로 개정 실험 산출물 작성
   - `UC`
   - `TEST-DESIGN`
   - `IMPL`
   - `TEST-EXECUTION`

3. 비교 항목 점검
   - `main_flow` 커버리지 차이
   - `preconditions` 커버리지 차이
   - `trigger` 커버리지 차이
   - runtime evidence 요구 반영 여부
   - validator의 누락 탐지력 차이

4. 결과 정리
   - 기존 방식 누락 항목
   - 개정 방식 추가 포착 항목
   - validator 판정 차이
   - 향후 문서 반영 필요성

### 4. 대체 검증 경로 B - 샘플 산출물 기반 시뮬레이션

조건:
- 실제 기존 사례가 당장 없는 경우

진행 방식:

1. 대표 요구사항 하나를 정한다.
   - 입력 상태
   - 시간 흐름
   - 관찰 가능한 상태 변화
   가 모두 있는 작은 예제가 적합하다.

2. 동일 요구사항에 대해 아래 두 세트를 작성한다.

기존 v1 방식:
- `UC`
- `IMPL`
- `TEST`

개정 실험 방식:
- `UC`
- `TEST-DESIGN`
- `IMPL`
- `TEST-EXECUTION`

3. 비교한다.
- 기존 `TEST`가 놓치는 시나리오가 있는가
- `TEST-DESIGN`이 추가된 뒤 누락이 구조적으로 드러나는가
- validator 기준으로 더 일관된 판정이 가능한가

### 5. 샘플 시뮬레이션에 필요한 최소 산출물 세트

기존 비교용:
- `UseCaseModelArtifact`
- `ImplementationArtifact`
- `TestReportArtifact`
- 필요한 경우 `ValidationReportArtifact`

개정 비교용:
- `UseCaseModelArtifact`
- `TestDesignArtifact`
- `ImplementationArtifact`
- `TestReportArtifact` (`TEST-EXECUTION`)
- 필요한 경우 `ValidationReportArtifact`

### 6. 성공 기준

아래가 충족되면 실험이 유효하다고 본다.

- 기존 방식에서 빠졌던 `main_flow` 단계가 `TEST-DESIGN`에 명시된다
- 기존 방식에서 빠졌던 `preconditions` 또는 `trigger`가 시나리오로 드러난다
- `runtime evidence required` 여부가 설계 단계에서 명확히 구분된다
- `TEST-EXECUTION`이 설계된 시나리오를 실제로 참조한다
- Validator가 기존보다 더 명확하게 누락을 `warning` 또는 `fail`로 판정할 수 있다

### 7. 다음 단계 선택지

현재 시점에서 가능한 다음 단계는 두 가지다.

#### 선택지 1 - 샘플 시뮬레이션 먼저 진행

적합한 경우:
- 실제 사례가 현재 작업공간에 없음
- 구조 변경의 실효성을 먼저 눈으로 확인하고 싶음

장점:
- 운영 문서 반영 전에 설계 품질을 검증할 수 있음
- 위험을 낮춤

#### 선택지 2 - 실제 문서 반영 먼저 진행

적합한 경우:
- 현재까지 초안 합의가 충분하고
- 샘플 시뮬레이션보다 문서 구조 고정이 더 시급함

장점:
- 문서/프롬프트/템플릿 정합성을 빠르게 맞출 수 있음

주의:
- 실제 효과 검증은 뒤로 밀리게 됨

### 8. 현 단계 권장안

- 현재 `workspace`에 실제 비교 사례가 없으므로, 다음 단계는 **샘플 산출물 기반 시뮬레이션을 먼저 진행**하는 것이 가장 안전하다.
- 즉시 문서 반영으로 가기보다, 작은 대표 요구사항 하나를 정해 기존 방식과 개정 방식 산출물을 비교해보는 것이 적절하다.
