# OpenSDLC Architecture Guidelines (제2계층 지침)

**제1장 아키텍처 및 통제 전략**

**제1조 (Strict Adherence, 엄격한 지시 준수)**

① (원칙) 모든 에이전트는 사용자가 요구한 사항 또는 직전 아티팩트에서 명시적으로 지시한 범위 내에서만 임무를 수행해야 한다.

② (적용) 에이전트는 자의적인 판단으로 UI 디자인, 아이콘, 레이아웃 변경 또는 기능 추가 등 요구사항에 없는 '창의적 최적화'를 수행해서는 아니 된다.

③ (예외) 보안상 치명적인 취약점이 발생하거나 코드의 물리적 실행이 불가능한 경우에 한하여, 해당 원인을 명시적으로 보고하고 필요 최소한의 방어적 수정을 가할 수 있다.

**제2조 (Full Traceability, 전방향 추적성 보장)**

① (원칙) 기획부터 구현, 테스트, 피드백에 이르는 산출물 파이프라인은 상호 연결 고리를 유지하여 추적성(Traceability)을 보장하여야 한다.

② (적용) 모든 아티팩트는 생성 시 반드시 그 근거가 되는 상위 아티팩트의 ID(`source_artifact_ids`)를 포함해야 하며, 테스트 항목은 요구사항 문서(UC)의 수용 기준(Acceptance Criteria)과 1:1로 맵핑되어야 한다.

③ (예외) 최초 요구사항 명세를 위해 생성되는 초기 UseCaseModelArtifact(UC) 등의 시발점 문서는 상위 참조를 생략할 수 있다.

**제3조 (Role-Based Orchestration, 역할 기반 오케스트레이션)**

① (원칙) 본 시스템은 기획, 개발, 테스트, 조정 등의 전문 영역을 개별 에이전트(PMA, Req, Code, Test, Coord, Val)에게 명확히 분리하여 할당한다.

② (적용) 특정 에이전트는 자신에게 부여된 역할의 바운더리를 넘어설 수 없으며, 타 에이전트의 고유 권한(예: 요구사항 기획 에이전트가 코드를 직접 구현하는 행위 등)을 침해할 수 없다. 

③ (예외) 시스템 구조 변경 작업 등 1개의 에이전트가 2개 이상의 역할을 대행하도록 사용자가 일시적으로 특권 권한을 부여한 경우 접근을 허가한다.

**제4조 (Validation Over Self-Assertion, 독립적 검증의 원칙)**

① (원칙) 에이전트가 산출한 결과물의 타당성과 규칙 준수 여부는 결과물을 생산한 에이전트 스스로의 선언(Self-Assertion)에 의존하지 않고, 물리적 환경과 논리적 구조를 기반으로 독립된 심판자(ValidatorAgent)에 의해 객관적으로 교차 검증되어야 한다.

② (적용) 아티팩트의 스키마, 추적성, 증거 채택, 상태 전이의 일관성은 항상 별개의 ValidatorAgent가 검증하며, 이 게이트를 통과(`Pass`)해야만 다음 프로세스로 이행할 수 있다.

③ (예외) 명백한 오작동 또는 논리적 모순 구조로 인하여 Validator 루프에 무한히 갇힌 경우, PMAgent 주도로 강제 중단 및 사용자 보고를 수행한다.

**제5조 (No-Shortcut Principle, 자율 수행 무생략 원칙)**

① (원칙) 반복적인 이터레이션을 자율적으로 진행하는 환경일지라도, 정해진 아티팩트 파이프라인 단계를 어떠한 이유로든 임의로 단축하거나 통합할 수 없다.

② (적용) 요구사항(UC) -> 검증(VAL) -> 테스트 설계(`TEST-DESIGN`) -> 검증(VAL) -> 구현(IMPL) -> 검증(VAL) -> 테스트 실행 검증(`TEST-EXECUTION`) -> 검증(VAL) -> 피드백(FB) -> 검증(VAL)으로 이어지는 물리적 파일 생성 흐름은 단 1회라도 생략될 수 없으며 각 단계마다 독립된 문서로 분리 생성되어야 한다.

③ (예외) 사용자가 직접 개입하여 특정 단계를 수동 승인으로 치환하는 경우 해당 이터레이션에 한하여 건너뛸 수 있다.
